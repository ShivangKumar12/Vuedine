# 0001 — ORM choice: Prisma over Sequelize

- **Status:** Accepted
- **Date:** 2026-06-09
- **Deciders:** Vuedine Engineering

## Context

We need an ORM for the Vuedine API. The two mainstream Node.js options are:

- **Prisma 5** — schema-first, generated client, Rust-backed query engine
- **Sequelize 6** — model-class-first, hand-written types, mature ecosystem

The decision affects every database call in the codebase, the migration workflow, type safety, and our scaling story (read replicas, multi-tenant query helpers).

## Decision

We pick **Prisma 5**.

## Rationale

| Criterion                 | Prisma 5                                                   | Sequelize 6                                         |
| ------------------------- | ---------------------------------------------------------- | --------------------------------------------------- |
| Type safety               | Generated types from schema, always in sync                | Manual typings, drift over time                     |
| Migration DX              | `prisma migrate dev` — declarative, diff-based             | `umzug` boilerplate, hand-written SQL               |
| Query API                 | Composable, predictable, single style                      | Inconsistent (raw vs. builder vs. instance methods) |
| Performance ceiling       | Rust query engine handles batching well                    | Mature, but slower for batch operations             |
| Read replica support      | First-class via `@prisma/extension-read-replicas`          | Manual `replication` config, error-prone            |
| Multi-tenant ergonomics   | Soft-delete + tenant filters via Prisma middleware (clean) | Hooks + scopes (workable but verbose)               |
| Raw SQL escape hatch      | `prisma.$queryRaw\`...\`` is parameterized + typed         | `sequelize.query()` returns `unknown[]`             |
| Community momentum (2026) | Active, fast-moving                                        | Maintenance-mode-ish                                |
| Studio / DX tooling       | Prisma Studio is excellent                                 | None equivalent                                     |

## Tradeoffs

1. **Prisma owns the migration process.** When DBA-controlled migrations are required (e.g., for a complex backfill), generate raw SQL with `prisma migrate diff --script`, edit by hand, then apply via `prisma migrate deploy`. Documented in Phase 2 of `BACKEND_PHASES.md`.

2. **Prisma's Node.js client size.** ~30 MB unpacked vs. Sequelize's ~5 MB. Negligible at our scale; matters only for serverless cold starts (which we don't deploy to).

3. **Prepared-statement incompatibility with PgBouncer transaction-mode pooling.** Mitigated by setting `?pgbouncer=true` in the connection string (Phase 8). Documented.

## Alternatives considered

- **Drizzle ORM** — newer, lightweight, growing fast. Rejected because the team has more Prisma experience and Prisma's migration tooling is more mature. Re-evaluate in 12 months.
- **Knex + raw SQL** — maximum control, minimum magic. Rejected because the team would re-implement type safety, migrations, and query composition from scratch.
- **TypeORM** — rejected; abandoned-feeling maintenance pace, decorator-heavy patterns cause friction with pure-JS preference.

## Consequences

- All database access goes through `src/db/prisma.js` (singleton client).
- Migrations are committed alongside schema changes; CI applies them via `prisma migrate deploy`.
- Soft-delete and tenant scoping are codified in Prisma middleware (Phase 2) so individual queries don't have to remember.
- ESLint rule blocks `$queryRawUnsafe` (Phase 9) to prevent SQL injection.
- A read replica is wired via a separate Prisma client (`src/db/prismaReplica.js`) used only for analytics/reports.
